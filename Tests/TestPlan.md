## 1. Введение

### 1.1 Назначение документа
Этот документ определяет общую стратегию, цели и подходы к тестированию кроссплатформенного приложения **TaskMaster AI**. Тестирование направлено на проверку соответствия продукта требованиям SRS, обеспечение стабильности основных функций и высокого качества пользовательского опыта.

### 1.2 Объекты тестирования
- **Клиентская часть (Frontend):** Приложение на React Native / Expo, включая UI, управление состоянием и взаимодействие с API.
- **Бэкенд-сервисы (BaaS):** Платформа Supabase, включая базу данных PostgreSQL, сервис аутентификации и серверные функции.
- **Интеграция с ИИ:** Взаимодействие с Google Gemini API через Edge Function.

---

## 2. Цели тестирования
- Проверить корректность реализации всех функциональных требований (аутентификация, CRUD задач, работа ИИ).
- Убедиться, что индикатор выгорания рассчитывается и отображается корректно.
- Проверить валидацию вводимых пользователем данных (например, обязательное название задачи).
- Подтвердить, что политики безопасности (RLS) в PostgreSQL работают и пользователь видит только свои данные.
- Оценить отзывчивость интерфейса и общее удобство использования.

---

## 3. Область тестирования

### 3.1 Функциональность для тестирования (In-Scope)
-✅ Регистрация / авторизация / выход.
-✅ Создание, редактирование, удаление и отметка о выполнении задач.
-✅ Расчет и отображение индикатора выгорания.
-✅ Создание и перенос задач с помощью команд ИИ-ассистента.
-✅ Навигация по недельному календарю.
-✅ Отображение статистики в профиле.

### 3.2 Функциональность вне тестирования (Out-of-Scope)
-❌ Нагрузочное тестирование API.
-❌ Тестирование в условиях отсутствия или очень медленного интернет-соединения.
-❌ Детальное тестирование на широком спектре физических устройств.
-❌ Автоматизированное UI-тестирование.

---

## 4. Стратегия тестирования
Основной подход — **ручное (manual) тестирование**, основанное на сценариях использования (Use Cases). Будут применяться как позитивные, так и негативные тест-кейсы для проверки устойчивости системы.

---

## 5. Виды тестирования

### 5.1 Функциональные
- **Smoke-тестирование:** Быстрая проверка работоспособности основных функций после каждого развертывания (вход, создание задачи).
- **Тестирование на основе Use Case:** Детальная проверка всех сценариев, описанных в документации.
- **Регрессионное тестирование:** Повторное выполнение ключевых тестов после внесения изменений для гарантии, что ничего не сломалось.

### 5.2 Нефункциональные
| Вид | Цель |
|:---|:---|
| **Удобство использования (Usability)** | Оценить, насколько интерфейс интуитивно понятен. |
| **Производительность (Performance)** | Замерить время отклика ключевых операций (загрузка, ответ ИИ). |
| **Безопасность (Security)** | Проверить, что пользователь не может получить доступ к чужим данным. |

## 6. Тестовое окружение
| Компонент | Среда |
|:---|:---|
| **Клиент** | Google Chrome (Web), Expo Go (iOS/Android) |
| **Бэкенд** | Проект в облаке Supabase |
| **База данных** | PostgreSQL (в облаке Supabase) |
| **ИИ** | Google Gemini API (с тестовым ключом) |

---

## 7. Тестовые данные
- **Пользователи:** Будут созданы несколько тестовых аккаунтов (например, `test1@example.com`, `test2@example.com`).
- **Задачи:** Набор задач с разной сложностью, важностью и датами для имитации реального использования.
- **Запросы к ИИ:** Набор стандартных и нестандартных команд ("создай задачу", "придумай план", "распредели нагрузку").

---

## 8. Оценка рисков

| Риск | Вероятность | Влияние | Стратегия смягчения |
|:---|:---|:---|:---|
| Нестабильная работа API Gemini | Средняя | Среднее | Реализована обработка ошибок; в будущем — добавить повторные попытки (retry). |
| Ошибки в расчетах на стороне БД | Низкая | Высокое | Тестирование сфокусировано на проверке результатов (шкала выгорания). |
| Рассинхронизация клиента и сервера | Средняя | Среднее | Использование механизма `refreshTasks()` после каждой мутации данных. |
